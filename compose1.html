Composing a Snapchat clone with MongoDB, Redis and Node.js (Part 1)

Share on Twitter Share on Facebook Share on Google+ Vote on Hacker News
Published Feb 16, 2016

There is no shortage of articles on the Internet that cover the process of inserting connection strings and dialing into an API or services, but there aren’t nearly as much that dwell on what a technology’s code or capabilities might look like outside of a vacuum. That’s warranted. As developers we’re the type of individual that button mashes ‘Enter’ as soon the video game tutorial loads, and I don’t blame you. Despite this, just like when you look up a video game walkthrough, sometimes we just want to want to see what the code looks like plugged in an app that isn’t merely a to-do list. This is also an especially convenient way to trial a certain DBaaS offering if you’re not ready to move your production data onto it just yet.

For this reason, I hacked a Snapchat clone over the weekend and open-sourced the code for us to dissect. Don’t worry, the code is scrappy, but it will be fun exercise for us to go over it and think of what I could have done better. You can try out the finished sample here. If you want to press a metaphorical ‘Enter’ button right now and start playing instead of reading for a bit, click here, or actually just press Enter.

Although this is more than the prototypical ‘Hello World’ app, we won’t dwell extensively on crucial but beyond the scope of this article matters such as input validation, CSS and IE8 hacks and migration to Angular 2.0 with Typescript.

If you’re not familiar with Snapchat, it’s an app with a simple concept and a colossal following. You add your friends and then you communicate with them primarily through pictures taken on your smartphone. The twist is that the pics or ‘snaps’ last only a few seconds, any integer amount from 1 to 10, and then you can never see them again. While there are handful of other features that build on Snapchat’s primary functionality, we’ll focus on what I just described for the sake of brevity and clarity.

Before we npm install anything, let’s quickly outline how our app will work. Just like the real thing, our open-source alternative, RedSnap, will allow users to log into an app, send friend requests to other users and then send and receive pictures. We’ll build a web client, but the API should be accessible from other devices with little modification as well. Our backend will need to store user info, sessions, pictures and metadata on the aforementioned. We’ll use node.js with express to facilitate prototyping and handle several concurrent connections with low processing requirements. Since we’re offering a web client, we’ll use Angular to offer a single page app experience to make up for the lack of a native phone app.

Compose offers a judicious amount of storage options for our data. If you were born and bred with Oracle, SQL Server or DB2, you won’t feel far from home with PostgreSQL, where as if you’re teenage relationships aligned to some extent with the successes and hiccups of Twitter, Pinterest and Buzzfeed, you might already be familiar with the likes of MongoDB and Redis. We won’t have any financial or critical data on our servers (the benefits of developing an ephemeral photo service) and we’re prototyping an MVP over the weekend, so we’ll forego calling in the cavalry with Postgres and design our application and data to work with MongoDB and Redis.

MongoDB is a document store. To put it simply, if you had a document, such as this blog post, we could store data as properties of the document. Properties of this blog post include author, date, title, text, hyperlink, etc. Some of our properties have properties of their own, for example, if there were comments on this blog, the comments could be a property of this blog and the comments could have their own properties such as date, author, text, and replies. We can store the entire document, the blog post, in collections which reside on the database instance. You can see how powerful this can be for prototyping and ease of development when you realize that you can store this entire article as a JSON object in MongoDB. We have to be careful not to duplicate our data however (e.g. store a user’s friend as a document in the user document), as this can result in us having to climb through trees of data to make alterations.

We can easily model our simple users as documents. They’ll have some basic user info, a username, a user id, an email, a hashed password and a salt for aforementioned password. We’ll also store references to the user’s friends. Friends come in two flavours, your current friends and users that request your friendship (which you must approve of). We’ll also store your total snap count, which consists of snaps you’ve sent and received. Finally, we’ll store the date that you joined RedSnap (so that we can annoy you when you’ve left your account idle for over a year... just kidding.)

When our users visit our site, they will log in to access data pertinent to them. We don’t want to have them log in every time they make a data request, so we store the fact that they’re perusing our site for the time being in a session. Since the session data is volatile and it’s only a minor inconvenience if the data is lost and the user has to log in again, we’ll favour Redis over MongoDB as a session store.

Redis is a key-value database. In brief, this means that we use a unique key to return a set of mapped data from the database. We can insert a key like a user id, and receive in return a list of fields and their values, or another key and receive a corresponding value. In the case of our sessions, we can input the key from the user’s cookie and receive data about when the user’s session expires, their username and whether they have access to certain parts of the site. Redis is an in-memory data store, meaning the entire thing is in RAM. As you’d expect, this makes Redis really fast and as touched upon in the last paragraph, convenient for volatile non-persistent data.

The most important data we’ll need to store for our application are undeniably the user’s pictures. These are sizeable items that need to be accessed asynchronously as not to tie the server up. So where do we store them?

We might consider a service like AWS S3 or Azure Blob Storage, but one of our goals for RedSnap as an open source project is vendor independence. Although we’d love for you to keep your Mongo and Redis instances on Compose, you can always move off if the need presents itself. With a proprietary service like S3 or Blog storage… not so much.

The answer in our cause is neither MongoDB or Redis; we’ll store the images on the filesystem. If we really wanted to, we could store the data in the database as base64 data URIs. With Redis, this might even be really fast and with MongoDB’s GridFS, we could store files larger than 16MB, but since our images won’t permanently reside on the file system, we can save costs and complexity by simply storing outside of the database. Our images range in size from ~500kb to 2 Mb. As you can imagine, this would quickly fill up a Redis instance which for our simple app would only consist of a few hundred megs of RAM. And we wouldn’t want to load large image data from MongoDB every time we query a document that might contain image data.

What we’ll do however, is store image metadata and file location data on Redis. Our users won’t need to query Redis for the images; they’ll access their metadata with keys, so this will make sure that the process of locating the images on the server is as speedy as possible.

Well that was a bit of a lecture wasn’t it? For some of you, this might have been the dreaded Far Cry 3 Blood Dragon tutorial. It’s tempting with a weekend project to just start vim and figure things as you go along, but it’s worth it to figure out some overarching logic for your app. In the wise words of Napoleon Hill: “Reduce your plan to writing. The moment you complete this, you will have given concrete form to intangible desire.”

Let’s explore this in the next portions of the tutorial where I detail my MongoDB and Redis code for RedSnap.
